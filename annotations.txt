Спецификация курсовой работы
Тема: «Справедливое мульти-ресурсное планирование в Kubernetes: реализация и исследование плагина Dominant Resource Fairness»

1. Цель работы

Разработать и исследовать плагин для стандартного планировщика Kubernetes (kube-scheduler), который реализует политику распределения ресурсов Dominant Resource Fairness (DRF). Плагин должен обеспечивать справедливое распределение двух типов ресурсов (CPU и память) между абстрактными пользователями, идентифицируемыми по меткам в подах.

2. Основные понятия

Пользователь (user): Абстрактная сущность, от имени которой запускаются поды. Идентифицируется по значению метки user в манифесте пода (например, user: "user-a").
Потребление пользователя (user consumption): Сумма запрошенных ресурсов (CPU, memory) всех подов данного пользователя, которые находятся в состоянии Running и были запланированы данным планировщиком.
Доминирующий ресурс (dominant resource): Тип ресурса, доля которого от общего объема кластера является максимальной для данного пользователя.
Доминирующая доля (dominant share): Значение этой максимальной доли (число от 0 до 1). Именно её выравнивание между пользователями является целью алгоритма DRF.

3.2. Детальное описание компонентов
scheduler/drf/state.go (Глобальное состояние)
Назначение: Хранит информацию о текущем состоянии кластера и пользователей.
Структуры данных:
type ClusterState struct: Основная структура.
mu sync.RWMutex: Обеспечивает конкурентный доступ.
TotalResources map[string]int64: Общие ресурсы кластера (CPU, memory), суммированные по всем нодам. Обновляется при старте или при изменении состава нод.
Users map[string]map[string]int64: Потребление каждого пользователя. Ключ внешней мапы — имя пользователя. Внутренняя мапа — тип ресурса -> потребленное значение.
Ключевые методы:
NewClusterState(client clientset.Interface): Конструктор. При создании:
Получает список всех подов через Kubernetes API.
Фильтрует поды, оставляя только те, у которых phase: Running и schedulerName: drf-scheduler.
Для каждого такого пода извлекает метку user.
Суммирует запросы ресурсов (requests) по CPU и памяти для каждого пользователя, инициализируя Users.
Получает список всех нод и суммирует их allocatable ресурсы в TotalResources.
AddUserConsumption(user string, resources map[string]int64): Увеличивает потребление пользователя (вызывается из Reserve).
RemoveUserConsumption(user string, resources map[string]int64): Уменьшает потребление пользователя (вызывается из Unreserve).
GetDominantShare(user string) float64: Возвращает текущую доминирующую долю пользователя. Рассчитывается как max(consumed[CPU]/total[CPU], consumed[Mem]/total[Mem]).

scheduler/drf/algorithm.go (Логика DRF)
Назначение: Содержит чистые функции для расчета DRF-метрики (dominant share). Не зависит от Kubernetes API.
Ключевые функции:
func CalculateDominantShare(consumed, totalResources map[string]int64) float64: Возвращает доминирующую долю пользователя как max(usedCPU/totalCPU, usedMem/totalMem).
scheduler/drf/drf.go (Плагин для Scheduler Framework)
Назначение: Интегрирует логику DRF в стандартный цикл планирования Kubernetes, используя точки расширения (extension points).
Структура:
type DRFPlugin struct { clusterState *ClusterState; handle framework.Handle }
Реализуемые методы (Extension Points):
Name() string: Возвращает имя плагина (DRFPlugin).
PreFilter(): Проверка глобальной справедливости.
Действие: Считает текущие доминирующие доли всех пользователей и максимальную доминирующую долю.
Логика допуска: Под допускается, если текущая доминирующая доля его пользователя строго меньше текущего максимума; при этом, если доминирующие доли всех пользователей равны, под допускается.
Примечание: В этом варианте плагин не вычисляет «какой станет доля» после добавления данного пода — сравнение выполняется по текущему состоянию потребления пользователей.
Результат: Если условие не выполняется, под помечается как Unschedulable на этом цикле и остается Pending до следующей переоценки (когда изменится состояние кластера).
Filter(): Проверка локальной выполнимости на ноде.
Действие: Проверяет, достаточно ли свободных ресурсов на конкретной ноде для размещения пода.
Логика: Сравнивает pod.Requests со свободными ресурсами ноды (NodeInfo.Allocatable - NodeInfo.Requested).
Результат: Пропускает только те ноды, где ресурсов достаточно.
Reserve(): Фиксация потребления.
Действие: Вызывается, когда под успешно прошел все фильтры и назначен на ноду.
Логика: Увеличивает счетчик потребления для пользователя в state.go на величину pod.Requests.
Unreserve(): Откат потребления (крайне важно для консистентности).
Действие: Вызывается, если после Reserve произошла ошибка (например, под не прошел более поздние проверки) или под был удален.
Логика: Уменьшает счетчик потребления для пользователя в state.go на величину pod.Requests. Предотвращает "утечку" ресурсов в статистике.
Score(): Ранжирование подходящих нод.
Действие: Выбирает "лучшую" ноду среди прошедших Filter.
Логика: headroom-score — предпочитает ноду, где после размещения пода останется больший запас CPU/памяти (0..100).
config/scheduler-config.yaml (Конфигурация)
Назначение: Говорит kube-scheduler, какой профиль использовать и какие плагины включить.




4. Ключевые нюансы и требования к реализации

Фиксированный набор ресурсов: Для простоты и наглядности работаем только с двумя ресурсами: cpu (измеряется в миллиядрах) и memory (в байтах). Это оговорено в постановке задачи.
Идентификация пользователя: Строго по label user в метаданных пода. Поды без такой метки должны либо отвергаться (с понятной причиной в событиях), либо относиться к системному пользователю unlabeled (это решение нужно принять и задокументировать).
Восстановление состояния (Reconciliation): При (пере)запуске планировщик обязан просканировать кластер и восстановить карту потребления (Users) на основе уже запущенных подов. Игнорирование этого шага приведет к "забыванию" старых подов и перекосам в справедливости.
Двухуровневая проверка ресурсов: Алгоритм DRF работает на уровне кластера, но планирование в Kubernetes происходит на уровне нод. Плагин должен комбинировать оба подхода:
PreFilter/Filter (глобальный): Не позволить запустить под, если его пользователь уже является (или остаётся) доминирующим по текущим долям в масштабах всего кластера, даже если ресурсы на нодах есть.
Filter (локальный): Не позволить запустить под на ноду, где недостаточно свободных ресурсов.
Потокобезопасность: scheduler/drf/state.go должен использовать мьютексы, так как планировщик обрабатывает несколько подов конкурентно.
Правильная обработка Reserve/Unreserve: Эта пара должна быть строго симметричной. Любое резервирование ресурса должно быть отменено в случае ошибки на следующих этапах.
Сравнение долей с плавающей точкой: Из-за погрешностей вычислений сравнение двух float64 должно производиться с использованием эпсилон (например, 0.0001).
Обработка ошибок и события (Events): Плагин должен создавать события Kubernetes (Events) для подов, которые он не может запланировать, с понятным сообщением (например, "DRF gate: user \"A\" dominant share 0.55 is not less than current max 0.55").


5. Поток обработки пода (Sequence Diagram) 

Пользователь создает под с schedulerName: drf-scheduler и лейблом user: "bob".
Запрос попадает в API Server.
kube-scheduler в профиле drf-scheduler запускает цикл планирования для этого пода.
PreFilter (DRFPlugin):
Плагин получает текущее состояние из state.go.
Считает текущие доминирующие доли и max-долю.
Если доля пользователя "bob" не меньше max (и доли не все равны), цикл прерывается, под остается Pending, создается Event.
Filter (стандартный + DRFPlugin):
Сначала выполняются стандартные фильтры (проверка портов, толерантсов и т.д.).
Затем вызывается Filter нашего плагина, который перебирает оставшиеся ноды и отсеивает те, где не хватит CPU или памяти для пода.
PreScore/Score (DRFPlugin):
Оставшиеся ноды ранжируются по headroom-score (остатку ресурсов после размещения pod).
Reserve (DRFPlugin):
Выбрана лучшая нода.
Плагин вызывает state.AddUserConsumption("bob", podRequest).
Bind (стандартный): Плагин bind назначает под на выбранную ноду.
PostBind (DRFPlugin): (Опционально) можно подтвердить успешное назначение.
(В случае ошибки после Reserve): Если бы на шаге 8 Bind не удался, фреймворк вызвал бы Unreserve для всех зарезервировавшихся плагинов, и наш плагин должен был бы вызвать state.RemoveUserConsumption(...), откатывая изменения.


6. Тестовые сценарии (workloads/)

Для демонстрации и исследования работы плагина предлагаются следующие сценарии:

Сценарий А (Баланс): Два пользователя запускают поды с одинаковыми запросами. Планировщик должен равномерно распределять ресурсы, не допуская, чтобы один занял все.
Сценарий Б (Дисбаланс по ресурсам): Пользователь A запускает много "CPU-емких" подов, пользователь B — много "Memory-емких". Планировщик должен стремиться к тому, чтобы их доминирующие доли были равны.
Сценарий В (Фрагментация нод): Создается ситуация, когда в кластере суммарно ресурсов достаточно, но они "разбросаны" по нодам так, что под большого пользователя не может быть запущен ни на одной ноде, хотя глобально кластер не перегружен. Планировщик должен корректно обработать это (запланировать позже или ждать освобождения ресурсов на одной ноде).
Сценарий Г (Восстановление после перезапуска): Во время работы планировщик перезапускается. После запуска он должен корректно подхватить текущее состояние кластера и продолжить справедливое планирование, как если бы перезапуска не было.
